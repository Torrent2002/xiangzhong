# 设计思考 · THINKING

> 这是「相中」这个方案的思考记录——为什么这么设计、做了哪些取舍、AI 的边界在哪、哪些是我刻意没做的。比代码本身更重要的是这些判断。

## 0. 一句话

「相中」想解决的是：相亲软件听不懂用户"说不清的喜欢"。我们的突破口是用 AI 把**无法被预设字段编码的偏好**变成可匹配的维度——而不是再给产品加几个筛选项。

---

## 1. 问题拆解：长尾需求的本质是"去 schema 化"

现有相亲软件只有两条路，都有天花板：

- **滑照片**（Tinder / 探探）：被动、低效、纯靠肉眼，无法主动筛选。用户要的是"帮我找"，得到的却是"你自己一张张看"。
- **填字段匹配**（Soul / 百合）：字段是平台**预设**的——年龄、身高、学历、收入、城市。覆盖不到"瓜子脸、不戴眼镜、书卷气、穿搭干净"这类偏好。

所以长尾需求的本质不是"字段不够多"，而是**偏好是 schema 外的、非结构化的**。再多加字段也填不满，因为用户的偏好本就不是离散标签能装下的。

> **判断**：真正的突破不是"加字段"，是"去 schema"——让用户用自然语言/图自由表达，AI 来理解。

---

## 2. 核心概念：偏好分两类，AI 各自击破

这是我这个方案的中心立论。用户偏好不是铁板一块，它有两种来源：

| | 显性偏好 | 隐性偏好 |
|---|---|---|
| **用户是否意识到** | 已明确 | 自己没发现 |
| **例子** | "不戴眼镜、瓜子脸、温柔文艺" | 嘴上说喜欢文艺，实际心动更多是高冷运动风 |
| **获取方式** | 自然语言 / 参考图 / 预设字段 | 行为信号（停留 / like·pass / 回看） |
| **AI 的作用** | 解析为**意图向量** | 反推为**偏好向量** |

两条向量汇入同一个匹配器；用户的每一次反馈同时更新这两类偏好。

**为什么这个分法重要**：现有产品只做显性偏好（而且做得很差，受限于 schema）。隐性偏好是被整个行业忽略的金矿——它对应的是"用户自己都不知道自己要什么"，这正是相亲里最高频的痛（"感觉不对"说不出理由）。谁先把隐性偏好做出来，谁就是下一代。

> **产品判断**：显性偏好让用户上手，隐性偏好让产品越用越懂你。后者是壁垒。

## 3. 关键设计取舍（产品向）

### 3.1 为什么是独立 app，而不是塞进小红书
- **场景聚焦**：相亲需要专注的心智状态，混在内容流里做不起来。
- **合规隔离**：相亲涉及强隐私与未成年保护，独立 app 便于做合规围栏，不连累主站。
- **不污染内容生态**：相亲行为若回流主站会扭曲推荐与社区氛围。

### 3.2 透明化匹配理由（刻意暴露 AI 判断）
每张推荐卡都写"为什么推荐给你：不戴眼镜 ✓ ｜ 瓜子脸 ✓ ｜ 唯一差异：她偏活泼"。

- **建信任**：黑盒推荐在相亲场景里用户不敢信（"是不是乱推的"）。透明化把 AI 的判断摊开。
- **给纠偏入口**：用户看到"唯一差异：偏活泼"，能立刻说"那我就要活泼的"，反馈闭环成立。
- **降误判伤害**：AI 错了用户能看出来，而不是被糊弄。

> 这是反"算法黑盒"直觉的取舍——多数平台藏算法，我们刻意露。因为相亲是高 stakes 决策，信任 > 神秘感。

### 3.3 隐性偏好的克制：可解释、可关闭
隐性偏好反推很容易做成"细思极恐"（"它怎么知道我喜欢这个"）。我的护栏：

- **主动提示而非偷偷改推荐**：用提示卡明确告诉用户"我发现你可能喜欢…"，让用户自己选要不要看，而不是悄悄把推荐流换掉。
- **可解释**：提示要带理由（"你最近 ♡ 的更多是高冷运动风"），不是凭空结论。
- **可关闭**：用户可拒绝并坚持原偏好。

> 价值观：让用户觉得"它懂我"，而不是"它在监视我"。

### 3.4 价值观护栏：品味匹配，不是外貌淘汰
外貌偏好是这产品的天然张力。我的处理：

- 把"外貌偏好"包装为**品味匹配**，而非**外貌淘汰**。
- **不展示被淘汰的人**（不出现"因为你不喜欢戴眼镜的所以推走了 XX"）。
- **不做单一颜值打分排名**（不搞"颜值 8.5 分"这种物化设计）。
- 种族、残疾等**不可作为筛选条件**，硬过滤。

> 这是产品能走多远的地基。外貌工具稍不注意就滑向物化，克制是必须的前置设计。

### 3.5 冷启动：借小红书生态 + 单边工具切入
新平台最大的敌人是双边冷启动。我的解法：

- **借力小红书公开笔记（合规授权）做初始 profile 池**，而不是从零拉用户填资料。
- **先做单边工具**："帮你找到你理想型的长相参考"——这本身就有人用，不依赖双边。用户用着用着，profile 池和偏好画像就建起来了，再过渡到双边匹配。
- Roadmap：单点工具 → 双边匹配 → 关系沉淀。

## 4. 技术取舍（技术向）

### 4.1 为什么是"结构化打分 + LLM 语义分"加权，而不是纯 LLM
纯 LLM 匹配有三个问题：**不可控**（每次结果漂移）、**成本高**（每用户每次匹配多次 LLM 调用）、**不可解释**（拿不出"为什么推荐"）。

所以 demo 用 `0.6 * 结构化分 + 0.4 * LLM 语义分`：
- 结构化分保证**可控、可解释、零成本兜底**。
- LLM 语义分补**bio 里那些结构化字段抓不到的气质**（"喜欢看展和旧书店"这种）。
- 权重可调，线上可 A/B。

### 4.2 双路架构：AI + 本地兜底，是产品决策也是工程决策
所有 AI 调用都有本地兜底（关键词解析 / 统计反推 / 模板理由）。

- **产品上**：相亲用户对"AI 抽风"零容忍，兜底保证下限体验。
- **工程上**：无 key / 限流 / 超时都能降级，演示不会白屏，线上不会雪崩。
- **商业上**：让产品在"有 key 高体验 / 无 key 也能用"间平滑，降低对新市场的冷启动门槛。

### 4.3 成本意识
每用户每次匹配的 LLM 调用要控制：

- **意图解析**：每用户每次输入 1 次。
- **语义打分**：对候选池每个 candidate 1 次——这是大头。降本思路：**意图向量缓存**（相似描述命中缓存）、**候选特征预计算**（bio 向量离线算好存库，线上只算用户意图 vs 候选向量的点积）、**批量打分**（一次 prompt 让 LLM 给多个 candidate 打分）。
- v1 demo 先不优化，但架构上预留（`ai_client` 统一封装便于加缓存层）。

### 4.4 数据与隐私
- **不存原始生物特征**：只存 AI 提取后的离散属性 + 向量，不存原始人脸。
- **行为反推用聚合信号**：统计的是"你 ♡ 的群体特征分布"，不是"盯着你看了谁"。
- **可一键删除**：偏好画像可导出、可清空。

### 4.5 为什么显性/隐性分两个模块
代码层面 `explicit_preference.py` / `implicit_preference.py` 分开，不是整洁癖，是因为：

- 两条链路的**迭代节奏不同**（显性偏稳定，隐性需要持续调模型）。
- **可独立灰度**（显性先上，隐性后上）。
- **可独立降级**（隐性反推挂了不影响显性匹配主链路）。

## 5. AI 的边界（我承认它做不到的）

诚实很重要，这几条是这方案的硬边界：

1. **审美主观且会变**：AI 给的是"概率匹配"，不是"正确答案"。今天喜欢温柔的，下个月可能想找活泼的。模型要能跟得上漂移——靠的就是隐性偏好的持续反推。
2. **多模态外貌提取有误判**：脸型、气质的视觉识别不会 100% 准，要人审兜底 + 用户反馈纠偏（透明化理由在这里起作用）。
3. **隐性偏好有"信息茧房"风险**：一直推用户 ♡ 过的类型，会越推越窄。解法是引入**探索性推荐**（偶尔推一个跳出画像的，看用户反应）——这是 roadmap 里必须做的，v1 没做，记在这里。
4. **AI 不能替代见面**：这是匹配工具不是相亲本身。预期管理要写进产品文案。

---

## 6. 局限性（这个 demo 没做到的）

不藏着：

- 候选池是 **mock 的 12 个**，真实产品需要双边真实用户。
- **头像是 SVG 插画**，没有真实照片识别（参考图通道置灰）。
- **行为反推是雏形统计**（♡ 的属性分布），不是真正的偏好向量建模。
- **无持久化**：行为日志进程内，刷新清空。
- **单城市/单人群**：没做地理与人群维度。

---

## 7. 如果继续做（roadmap）

1. **真实多模态参考图**：上传理想型照片 → vision 模型提取特征 → 走同一匹配器。
2. **更细行为建模**：停留时长、滑动速度、反复回看、消息回复延迟，喂给偏好向量模型。
3. **探索性推荐破茧房**：ε-greedy 式地偶尔推画像外的候选。
4. **双边匹配 + 关系沉淀**：从"找人"走到"匹配成功后的轻社交"。
5. **意图向量缓存 + 候选向量离线库**：把单次匹配的 LLM 调用从 O(候选数) 降到 O(1)。

---

## 8. 我的思考：范式转变才是真突破

回头看，这个题最容易掉进的坑是"用 AI 再多加几个筛选项"——比如加个"脸型"字段让用户选。那是 schema 内的小修小补，不叫突破。

真正的突破点是把相亲匹配的范式从**标签检索**升级成**语义 + 视觉向量匹配 + 行为反推**：用户不用再把自己塞进平台的字段框里，而是自由地说、自由地看，AI 去理解。同时把"用户自己都没意识到的偏好"这个被全行业忽略的需求，第一次变成产品功能。

如果只能让面试官记住一句话，是这句：**"不是加字段，是去 schema；不是猜你想要，是发现你其实想要。"**
