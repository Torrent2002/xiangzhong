# 相中 · Xiangzhong — 项目改进意见

> 针对「相中」相亲匹配 demo 的工程改进清单。核心判断：demo 产品点子好（显性/隐性偏好双通道），但工程上停在"能跑"没到"可靠"——可靠性、输出质量、异步范式、数据评估、隐私合规都需补。
>
> 用 C 端开发思维审视：**默认会挂 + 用户没耐心 + 数据驱动**。所有改进从这三条推导。

---

## 一、改造总表（8 主题）

| # | 主题 | 现状问题 | 改法 | 优先级 |
|---|---|---|---|---|
| **A** | **可靠性套件** | 降级链断在"有 key 失败"；无超时/重试/熔断/限流 | 超时 + 重试分类(瞬态重试/永久不重试) + 退避 + 降级链(有 key 失败也兜底) + 熔断 + 限流 | 🔴必做 |
| **B** | **输出质量** | 无 schema 校验、无幻觉防护、无 temperature 控制 | 结构化输出(JSON schema) + 校验失败重试 + `temperature=0` + 透明化理由可回溯到真实特征(防编造) | 🔴必做 |
| **C** | **异步+延迟分层** | LLM 疑似同步阻塞、无流式、无离线特征 | 离线算候选 embedding → 向量库；在线拆"ANN 出卡(<100ms)+流式透明化"；行为准实时异步更新偏好 | 🔴必做 |
| **D** | **行为日志+评估** | 进程内刷新丢；隐性偏好反推无评估方法 | 行为日志落 SQLite；拿 12 候选预置标签做 ground truth 算反推命中率 | 🔴必做 |
| **E** | **C 端体验** | 固定 3 步脚本、无错误态、无分页、移动端未明 | 骨架屏 + 乐观 UI + 错误兜底卡 + 候选分页 + 响应式确认 | 🟠建议 |
| **F** | **工程化** | 单进程、无健康检查/日志/Docker、API key 管理不明 | +`/health` + 结构化日志 + Dockerfile；明确 key 不下发前端(走后端代理) | 🟠建议 |
| **G** | **安全隐私** | 相亲敏感数据零合规、行为日志未告知 | 加隐私说明 + 数据本地化 + 行为日志可清除 + 告知 | 🔴必做 |
| **H** | **README** | 缺工程判断开场、局限未分级、无截图、技术栈薄 | 加"LLM 三难 + 如何应对"开场；局限拆"取舍/缺口"两栏；加截图 GIF | 🟠建议 |

## 二、延迟分层（C 主题核心，面试必讲）

这个 demo 本质是延迟敏感的 LLM 服务，和小红书 Feed 是同构问题（候选人=笔记要离线算特征，意图=query 线上算，行为=画像准实时更新）。延迟分四层：

| 层 | 延迟 | 干什么 |
|---|---|---|
| 离线 | 分钟~小时 | 候选照片多模态特征 + 简介 embedding → 向量库 |
| 准实时 | 秒级 | 行为信号 → 异步队列 → 偏好向量更新 |
| 在线-重 | 百 ms~秒 | 意图解析(LLM)、透明化理由(LLM，流式) |
| 在线-轻 | <100ms | 向量召回(ANN)、打分、Top-N |
| 感知 | 首字<1s | 流式输出、骨架屏 |

**原则：离线能算的绝不上线，线上重活用流式掩盖。**

## 三、优先行动（时间有限时）

**第一档（必做，性价比最高）：**
1. **A+B 可靠性 + 输出质量** → demo 从"能跑"变"可靠且输出可信任"
2. **C 离线 embedding + 在线拆快慢 + 流式** → 解决"用户等不了"
3. **D 行为日志持久化 + 隐性偏好评估** → 把创新点从口号变可量化
4. **G 隐私说明** → 相亲场景信任底线

**第二档（加分）：** E 体验、F 工程化、H README。

## 四、落到 demo 文件的具体改造点

| demo 文件 | 现状 | 改造 | 对应主题 |
|---|---|---|---|
| `modules/ai_client.py` | 同步调用，仅 key 有无两态降级 | 加超时 + 流式 SSE + 失败降级兜底 + 重试分类 | A、C |
| `modules/explicit_preference.py` | 同步出结果 | 拆成"先 ANN 出卡 + 骨架"和"流式透明化"两段 | C、E |
| `modules/implicit_preference.py` | 进程内日志 | 行为日志落 SQLite + 异步队列更新偏好向量 | C、D |
| 新增 `modules/candidate_indexer.py` | 候选特征无预计算 | 入库 → 离线 embedding → 向量库 | C |
| 新增 `eval/preference_eval.py` | 无评估 | 用候选预置标签做 ground truth 算隐性偏好命中率 | D |
| `app.py` | 单进程无运维 | +`/health` + 结构化日志 + Dockerfile | F |
| 前端 `static/` | 无反馈态 | 骨架屏 + 乐观 UI + 错误兜底卡 | E |
| `README.md` | 直接讲"是什么" | 加 LLM 三难开场 + 局限分级 + 截图 | H |
| 全局 | 零隐私表述 | 加隐私说明 + 行为日志告知 + 可清除 | G |

## 五、面试故事线（改造完怎么讲）

> 「这个相亲 demo 最初只做到『能跑』。后来我用 C 端思维重审，发现它本质是个延迟敏感的 LLM 服务——LLM 调用慢、输出随机、还会幻觉配错人。于是我从三方面补：可靠性上补全降级链（有 key 失败也兜底）+ 重试分类 + 超时；输出质量上强制结构化 + 校验 + 幻觉防护，理由可回溯到真实特征防编造；异步上把候选人特征提取挪到离线、在线拆成"快速出卡 + 流式透明化"、行为信号准实时更新偏好，并定义了离线/准实时/在线/感知四层延迟预算。最终 demo 具备了『可观测、可恢复、可降级、可量化』的工程闭环。」

| 小红书 Feed | 相亲 demo | 本质 |
|---|---|---|

| 用户搜索 query | 用户自然语言描述意图 | 线上实时算向量 |
| 用户行为画像 | ♡/✕/停留 → 偏好向量 | 离线/准实时反推 |
| "相关推荐" | "你可能其实喜欢…" | 召回+透明化 |
